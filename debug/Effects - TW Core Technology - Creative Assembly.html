    Effects

    • Created by Peter Ellsum (CA), last modified on Mar 23, 2023

    Introduction

    Effects are a generic system for distributing modifiers between game entities.
    These modifiers range from RPG style stat modifiers (i.e, +10 to Attack Power) to data sources used to communicate between different gameplay systems (i.e, Economic income and expenditure, Diplomatic Attitudes).


    The new system has been written to be extensible, performant, and traceable.  It should be possible for new systems to provide effects to the system, or to distribute effects through a complex scope, by implementing simple high level modules, without any changes to the underlying effects system.

    Components

    EffectSource

    An EffectSource component contains the collated set of effects that are distributed from an entity.  For example, a character would collect all of their effects from their Skills, Equipment, Traits, etc, and store them in the EffectSource.

    EffectDistributor

    An EffectDistributor is a middle level in between EffectSources and EffectReceivers. It collects effects from all EffectSources that fall within its scope. These effects are then forwarded to EffectReceivers that fall within their scope. 
    The distributor is a necessary intermediary, as directly connecting EffectSources to EffectReceivers would cause expontential growth in linkages. For example, having ten characters distributing to each other would generate one hundred connections, where as using a distributor would only generate twenty connections.

    EffectReceiver

    An EffectReceiver is the end point of effects. Effects are forwarded from distributors and collected into the receiver.

    Modifiers (i.e +10 Attack, +2 Income/turn) are combined with duplicates here and can be queried through a simple interface for use in game systems.

    Systems

    EffectProvision

    Effect Provision is responsible for forwarding effects from Effect Providers to Effect Sources. This process operates on a "tag and sweep" basis where Effect Sources tagged for reprocess will be collected and reprocessed, so only a subset of entities will be processed in a single iteration of the system.

    Effect Provision makes use of a plugin style architecture for allowing higher level systems to participate within the provision step. Any higher level system can implement the IEffectProviderModule interface and be registered with the system.

    EffectDistribution

    Effect Distribution is responsible for both forwarding effects from Effect Sources to Effect Distributors and forwarding effects from Effect Distributors to Effect Receivers. This process operators on a "tag and sweep" basis where Effect Distributors tagged for reprocess will be collected and reprocessed, so only a subset of entities will be processed in a single iteration of the system.

    Effect Distribution makes use of a plugin style architecture for allowing higher level systems to participate within the distribution step. Any higher level system can implement the IEffectDistributorModule interface and be registered with the system.

    Combined Diagram

    Provider modules can provide effects to any number of sources. They are modules at the system level and do not store per-entity data.

    Sources can forward effects to any number of distributors, they can even forward to the same distributor multiple times under different effect scopes.

    Distributors can forward effects to any number of receivers, they can even forward to the same receivers multiple times under different effect scopes.

    Receivers are completely unaware of the source/distributor graph below them, but each individual effect contains traceback information regarding which Provider/Source pair the effect originated from.

    Provider Interface

    For a system to provide effects it must implement the IEffectProviderModule interface and register itself with the system. The inner workings of the module can be as complex as desired, but note systems and modules should not store state.


    Interface

    ?
    class IEffectProviderModule
    {
    public:
        virtual ~IEffectProviderModule() {}
     
        //Data definition of the provider. Used to uniquely identify the provider and bind it to the game
        virtual const EffectProviderDef& definition() const = 0;
     
        //Use this provider to push effects into the specified effect source.
        virtual void provide_effects(const ecs::World& world, ecs::Entity effect_source_id, EffectSource& effect_source) const = 0;
    };

    The effect provider definition should be a Jade record used to uniquely identify this module to consuming systems.

    Beyond this the effect provider should simply add any effects the module provides to the source in the "provide_effects" method. Full read access is granted to the entire world, so any complex conditions desired can be implemented here.

    Example

    ?
    struct TestEffects
    {
        ECS_COMPONENT(TestEffects);
        ScopedEffectsCollection entity_effects;
    };
     
    struct TestEffectProvider final
        : public Effects::IEffectProviderModule
    {
        virtual const EffectProviderDef& definition() const override
        {
            static const EffectProviderDef c_def{ "Test Effects" };
            return c_def;
        }
     
        virtual void provide_effects(const ecs::World& world, ecs::Entity effect_source_id, EffectSource& effect_source) const override
        {
            const EffectProviderDef& provider_def = definition();
            if (TestEffects* const test_effects = world.get_component<TestEffects>(effect_source_id))
            {
                for (const ScopedEffects& scope_effects : test_effects->entity_effects)
                {
                    effect_source.add_effects(provider_def, scope_effects);
                }
            }
        }
    };

    In the example above a component is used to store a simple effect list on a per-entity basis. Then the provider simply attempts to retrieve this component on the target entity and forward the effects stored within to the effect source.

    Distributor Interface

    For a system to distribute effects it must implement the IEffectDistributorModule interface and register itself with the system. The inner workings of the module can be as complex as desired, but note systems and modules should not store state.


    Interface C++

    ?
    class IEffectDistributorModule
    {
    public:
        virtual ~IEffectDistributorModule() {};
     
        //Scope that this distributor operates at. A data driven identifier to bind the module to the game
        virtual const EffectScope& scope() const = 0;
     
        //Calculate the distributor entity for an effect source (an entity that *sends* effects to other objects)
        virtual ecs::Entity calculate_distributor_for_effect_source(const ecs::World& world, ecs::Entity source) const = 0;
     
        //Calculate the distributor entity for an effect target/sink (an entity that *receives* effects to other objects)
        virtual ecs::Entity calculate_distributor_for_effect_target(const ecs::World& world, ecs::Entity target) const = 0;
     
        //Calculate a complete list of entities that provide effects through the given distributor scope. Returns true if any entities were found
        virtual bool calculate_sources_for_distributor(const ecs::World& world, ecs::Entity distributor, CA_STD::VECTOR<ecs::Entity>& result) const = 0;
     
        //Calculate a complete list of entities that receive effects through the given distributor scope. Returns true if any entities were found
        virtual bool calculate_targets_for_distributor(const ecs::World& world, ecs::Entity distributor, CA_STD::VECTOR<ecs::Entity>& result) const = 0;
     
        //Test if a target entity is valid to receive effects from a given source entity
        virtual bool is_target_valid_for_source(const ecs::World& /*world*/, ecs::Entity /*source*/, ecs::Entity /*target*/) const { return true; }
    };
    • Effect Scope - Data identifier for the scope that the distributor module operates at. i.e "AreaCharacters", "FactionWhoseNameStartsWithAAndEndsWithY"
    • Calculate Distributor for Effect Source - Given an effect source, calculate which entity would act as the distributor. This may not return a valid entity if the effect source is in an invalid state for the distributor to apply
    • Calculate Distributor for Effect Receiver - Given an effect receiver, calculate which entity would act as the distributor. This may not return a valid entity if the effect receiver is in an invalid state for the distributor to apply.
    • Calculate Sources for Distributor - Given an effect distributor, calculate which entities would act as sources. Returns true if any sources were found.
    • Calculate Receivers for Distributor - Given an effect distributor, calculate which entities would receive effect from the distributor. Returns true if any receivers were found.
    • Is Receiver Valid for Source - Given an effect source and an effect target, determine if the receiver should receive effects from the source. This allows secondary filtration of receivers, such as filtering for diplomatic state.

    Example C++

    ?
    struct DistributorFactionOwnCharacters final
        : public IEffectDistributorModule
    {
        const EffectScope& scope() const
        {
            static const EffectScope c_scope{ "Owned Characters" };
            return c_scope;
        }
     
        ecs::Entity calculate_distributor_for_effect_source(const ecs::World& world, const ecs::Entity source) const override
        {
            return DistributorFactionOwnCharactersUtil::calculate_faction_for_entity(world, source);
        }
     
        ecs::Entity calculate_distributor_for_effect_receiver(const ecs::World& world, ecs::Entity target) const override
        {
            if (world.get_component<Character>(target))
            {
                return DistributorFactionOwnCharactersUtil::calculate_faction_for_entity(world, target);
            }
            return ecs::INVALID_ENTITY;
        }
     
        bool calculate_receivers_for_distributor(const ecs::World& world, const ecs::Entity distributor, CA_STD::VECTOR<ecs::Entity>& result) const override
        {      
            const card32 prev_result_size = result.size();
            if (world.get_component<Faction>(distributor))
            {
                world.iterate([distributor, &result](const ecs::Entity entity, const Character*, const EffectReceiver*, const Owner* owner)
                    {
                        if (owner->owner_id == distributor)
                        {
                            result.push_back(entity);
                        }
                    });
            }
            return result.size() != prev_result_size;
        }
     
        bool calculate_sources_for_distributor(const ecs::World& world, ecs::Entity distributor, CA_STD::VECTOR<ecs::Entity>& result) const override
        {
            if (world.get_component<Faction>(distributor))
            {
                result.push_back(distributor);
                world.iterate([distributor, &result](const ecs::Entity entity, const Owner* owner)
                    {
                        if (owner->owner_id == distributor)
                        {
                            result.push_back(entity);
                        }
                    });
                return true;
            }
            return false;
        }
    };

    The above example distributes from anything owned by a faction to any characters owned by a faction.

    Note: The interfaces around ownership are still in very early development and this will likely be easier and less verbose later in development.

    Interface C#

    Distributors may also be implemented in C# by types that implement the IEffectDistributionModule interface.

    Note: Distributors must be pure and not mutate world state in any of their methods.

    ?
    public interface IEffectDistributionModule
    {
        EffectScope Scope { get; }
        Entity CalculateDistributorForEffectSource(EntityWorld world, Entity source);
        Entity CalculateDistributorForEffectReceiver(EntityWorld world, Entity target);
        IEnumerable<Entity> CalculateSourcesForDistributor(EntityWorld world, Entity distributor);
        IEnumerable<Entity> CalculateReceiversForDistributor(EntityWorld world, Entity distributor);
        bool IsReceiverValidForSource(EntityWorld world, Entity source, Entity target);
    }

    Example C#

    ?
    public class TestEffectDistributor
    {
        static readonly EffectScope s_Scope = Data.Get<EffectScope>("TestEffectScope");
     
        EffectScope Scope => s_Scope;
        Entity CalculateDistributorForEffectSource(EntityWorld world, Entity source) => source;
        Entity CalculateDistributorForEffectReceiver(EntityWorld world, Entity target) => target;
        IEnumerable<Entity> CalculateSourcesForDistributor(EntityWorld world, Entity distributor) => distributor.ToEnumerable();
        IEnumerable<Entity> CalculateReceiversForDistributor(EntityWorld world, Entity distributor) => distributor.ToEnumerable();
        bool IsReceiverValidForSource(EntityWorld world, Entity source, Entity target) => true;
    }

    Formulas

    Effect formulas can be defined in jade, at the time of writing they are relatively simple. In the future we intend to improve this functionality and allow more control, but the actual internal mathematic formula is currently fixed. Formulas allow you to define a set of modifiers and how they mutate a base value, they are applied as in the formula below. This does mean that multipliers and precentile modifiers will have no effect if the base value is zero.


    Value = Base + Additive + (Base x Multiplicative) + (Base x Percentile 100)

    Supported Operators

    • Additive - Simply adds the value of the modifier to the formula result
    • Multiplicative - All multiplicative values are summed together and then the base value is multiplied by the sum and added to the result.
    • Percentile - All percentile values are summed together and then the summed percentage of the base value is added to the formula result.

    Future Work

    Allow modifiers to be retrieved from scopes other than the local object. Defined as a scope/modifier pair.

    Allow caps for contributions from specific modifiers

    Allow designers to specify the order of operations

    Node graphs


    Data

    Data for effects is exclusively stored and edited in Jade. Unlike previous versions of the effect system, designers are free to add new modifiers (for use in formulas) and effect scopes (for implementation in C#) without support from the programming team, otherwise the structure should be familiar to any users of the previous effect system.


    Structures

    • Effect Modifier ID - Identifier for a single modifier value. Previously called a bonus value ID. This is the key that identifies a specific value. Designers are free to add new modifiers, no code changes required.
    • Effect Definition - Effect definitions are a root level object in Jade. This means that each effect definition will be it's own file and that they may be reused in many effect lists.
    • Effect Scope - Identifier for an effect scope. Used to bind effect data to code (C++ or C#) that controls scopes.
    • Scoped Effects - Data structure embedded into higher level concepts (such as character skills, equipment, technology, etc.). Defines a list of effects and the scope they should be distributed across.
    • Effect Formula - Data structure embedded into higher level concepts (such as unit stat modifiers, building cost, recruitment caps, etc.). Defines lists of modifiers and they operation by which they are attributed to the formula calculation. See the formula section for more information.

    FAQ

    Who is this document for?

    Everybody

    How do I make a new feature provide effects?

    Implement a new effect provider. (See the Provider Interface section)

    Register the effect provider with the EffectProvision system.

    How do I make a new distribution scope for effects?

    Implement a new distributor. (See the Distributor Interface section) 

    Register the distributor with the EffectDistribution system.
